# Snort_Rule_Header_IP_And_Ports_Mental_Model

| Concept | Operational Meaning |
|---|---|
| Rule header structure | action, protocol, source_ip, source_port, direction_operator, destination_ip, destination_port |
| Source IP | The IP address or range to the left of the direction operator; represents where traffic originates |
| Source port | The port to the left of the direction operator; represents the originating port |
| Destination IP | The IP address or range to the right of the direction operator; represents where traffic is going |
| Destination port | The port to the right of the direction operator; represents the target port |
| -> operator | Unidirectional; rule applies only to traffic flowing from source to destination as written |
| <> operator | Bidirectional; Snort evaluates the IP and port pairs in either direction; useful for capturing both sides of a conversation |
| any keyword | Wildcard; matches any IP address or any port when used in the IP or port position |
| ! negation operator | Excludes the specified IP or port from matching; placed directly before the value with no space |
| CIDR notation | Used to match a range of IPs; example: 192.168.100.0/24 matches all IPs in that subnet |
| Single host match | A specific IP with no CIDR suffix matches only that exact host |
| Negation scope | The ! operator applies only to the single value or CIDR block immediately following it |
| Negation vs CIDR | !192.168.100.211 excludes one host; 192.168.100.0/24 matches the whole subnet; these are not interchangeable |

# Snort_Rule_Header_IP_And_Ports_Configuration_Checklist

| Step | Task | Device | Command | Expected Result |
|---:|---|---|---|---|
| 1 | Open the Snort local rules file | Ubuntu Workstation | sudo nano /etc/snort/rules/local.rules | Rules file opens for editing |
| 2 | Write the action and protocol fields | Ubuntu Workstation | alert icmp | First two fields of the rule header are set |
| 3 | Set the source IP to a specific host | Ubuntu Workstation | alert icmp 192.168.100.211 | Rule will match traffic originating from that host only |
| 4 | Set the source IP using negation to exclude a host | Ubuntu Workstation | alert icmp !192.168.100.211 | Rule will match all traffic except from that host |
| 5 | Set the source IP using CIDR to match a subnet | Ubuntu Workstation | alert icmp 192.168.100.0/24 | Rule matches any host within that subnet |
| 6 | Set the source port to any | Ubuntu Workstation | alert icmp !192.168.100.211 any | Source port field added; any means all ports |
| 7 | Add the unidirectional operator | Ubuntu Workstation | alert icmp !192.168.100.211 any -> | Traffic direction is set from source to destination |
| 8 | Set the destination IP to any | Ubuntu Workstation | alert icmp !192.168.100.211 any -> any | Rule applies to all destination IPs |
| 9 | Set the destination port to any | Ubuntu Workstation | alert icmp !192.168.100.211 any -> any any | Full header is complete; rule options follow in parentheses |
| 10 | Add rule options to complete the rule | Ubuntu Workstation | alert icmp !192.168.100.211 any -> any any (msg:"ICMP PACKET DETECTED"; sid:1000002;) | Complete valid rule is written |
| 11 | Replace -> with <> to capture bidirectional traffic | Ubuntu Workstation | alert icmp 192.168.100.211 any <> any any (msg:"ICMP BIDIRECTIONAL"; sid:1000003;) | Rule matches traffic in both directions between the defined endpoints |
| 12 | Save the rules file | Ubuntu Workstation | Ctrl+O, Enter, Ctrl+X | File is saved and editor exits |
| 13 | Test Snort configuration syntax | Ubuntu Workstation | sudo snort -T -c /etc/snort/snort.conf | Snort reports configuration OK with no errors |
| 14 | Run Snort and generate ICMP traffic from the excluded IP | Win_Station | ping <target_ip> from 192.168.100.211 | No alert fires because that host is excluded by negation |
| 15 | Generate ICMP traffic from a different host | Ubuntu Workstation | ping <target_ip> | Alert fires because source IP is not the excluded host |

# Snort_Rule_Header_IP_And_Ports_Skeleton

Unidirectional rule skeleton
<action> <protocol> <src_ip> <src_port> -> <dst_ip> <dst_port> (msg:"<LABEL>"; sid:<SID>;)

Bidirectional rule skeleton
<action> <protocol> <src_ip> <src_port> <> <dst_ip> <dst_port> (msg:"<LABEL>"; sid:<SID>;)

Single host match
alert icmp 192.168.100.211 any -> any any (msg:"ICMP FROM SPECIFIC HOST"; sid:1000004;)

Negation to exclude a single host
alert icmp !192.168.100.211 any -> any any (msg:"ICMP NOT FROM EXCLUDED HOST"; sid:1000005;)

Subnet match using CIDR
alert icmp 192.168.100.0/24 any -> any any (msg:"ICMP FROM SUBNET"; sid:1000006;)

Bidirectional conversation capture
alert tcp 192.168.100.211 any <> any 80 (msg:"HTTP CONVERSATION"; sid:1000007;)

# Snort_Rule_Header_IP_And_Ports_Failure_Checks

| Symptom | Command | What Usually Broke |
|---|---|---|
| Rule triggers on the excluded IP | cat /var/log/snort/alert | The ! operator was not placed directly before the IP with no space; check rule syntax |
| Rule never triggers on any traffic | sudo snort -T -c /etc/snort/snort.conf | Direction operator is wrong or source/destination fields are mismatched with actual traffic flow |
| Snort reports syntax error on startup | sudo snort -T -c /etc/snort/snort.conf | Missing space between fields, incorrect CIDR notation, or invalid character in IP field |
| Bidirectional rule only triggers one way | Review rule and confirm <> operator is used | Rule uses -> instead of <>; replace with <> to capture both directions |
| Subnet rule matches the wrong hosts | Review CIDR block in rule | CIDR prefix length is incorrect; verify the subnet mask covers only the intended range |
| Rule matches all traffic instead of the target subnet | cat /var/log/snort/alert | The any keyword was used instead of the intended CIDR block |
