# Splunk_Brute_Force_Success_Verification_Mental_Model

| Concept | Operational Meaning |
|---|---|
| `action` field | Normalized Splunk field that captures the outcome of an event — common values are `success`, `failure`, `blocked`, `allowed` |
| `action=success` on syslog | Filters syslog events to only those where authentication succeeded — confirms a valid credential was found |
| Attack termination indicator | Brute force typically stops when valid credentials are obtained — a success event at the end of a failure sequence marks the transition |
| Successful login as attack pivot point | A successful authentication is what enables the attacker to move from external probing to internal access — the prerequisite for the data exfiltration observed in Tasks 1–4 |
| Attack chain completion | Brute Force (Task 6) → Successful Login (Task 7) → Data Exfiltration (Tasks 1–4) forms the complete attack sequence in chronological order |
| Last page / last attempts inspection | The lab instructs review of the final log entries — brute force tools stop issuing attempts once a valid credential is confirmed, making the success event appear near the end of the sequence |

# Splunk_Brute_Force_Success_Verification_Failure_Checks

| Symptom | Likely Cause | Fix |
|---|---|---|
| No results for `action=success` | `action` field not normalized or named differently in this dataset | Remove `action=success` and inspect `_raw` field on final log entries directly for success keywords |
| Results present but success event not clearly identifiable | Raw log text requires manual reading | Sort by `_time` ascending and navigate to the last entries — success appears at the end of the failure sequence |
| Conflicting conclusion — no success found | Time range too narrow, cutting off final log entries | Confirm All Time is set; paginate to the last result page |
| Temptation to conclude failure because exfiltration was on a different IP | Exfiltration src IP (`172.16.100.21`) is the internal host the attacker accessed after logging in — not the attacker's own IP | The successful login gave the attacker control of `172.16.100.21`, which then performed the exfiltration |
