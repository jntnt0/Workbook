# Lab 9 — Tracing Lateral Movement via File Server Propagation

## Tab 1

Lateral movement is how an attacker turns a single compromised host into a network-wide problem. One of the oldest and most effective techniques is shared drive propagation — copying malware onto a file server that other hosts on the network access regularly. Every machine that connects to the share and executes the malicious file becomes another infected host, all without the attacker needing to touch each machine individually. The infection scales itself.

The goal of this task is to identify the specific command used by `GoogleUpdate.exe` to copy itself to the file server, confirm the destination as a shared drive, and trace how the malware reached the second infected host identified in Lab 8.

---

### Step-by-Step Thought Process

**Return to the GoogleUpdate.exe command table.**  
The full command history of the malicious parent process — built in Lab 2 — contains every action the malware took, including file operations. Within that table, one CommandLine entry stands out: a copy operation targeting an IP address in the Server Segment.

**Identify the copy command.**  
One of the CommandLine rows shows the malware copying an instance of `GoogleUpdate.exe` from the local temp or Downloads directory to `192.168.200.6`. This is not a command execution — it is a file write operation pushing the payload onto a remote host.

**Confirm the destination as a shared drive.**  
Cross-referencing `192.168.200.6` against the network diagram places it in the Server Segment (`192.168.200.X`). The destination path in the copy command includes a UNC share path — confirming this is a shared drive accessible to other hosts on the network, not a private server endpoint.

**Trace the propagation mechanism.**  
Once the malicious binary lands on the shared drive, any host that accesses that share and executes the file becomes infected. The attacker does not need to interact with each host individually — the shared drive does the distribution automatically through normal user and system activity.

**Eliminate the other options.**  
- SSH propagation would appear as outbound SSH connection events with key-based or credential-based authentication — none of that appears in the logs.
- Browser download on other hosts would require separate Chrome and Outlook process chains on each machine — not present.
- Single host claim is disproven by the copy command and the two-host count confirmed in Lab 8.

### Expected Splunk Query

```
source=/home/splunk/data.csv ParentImage="C:\\Users\\User067\\Downloads\\GoogleUpdate.exe"
| table ParentImage CommandLine
```
Scan the CommandLine column for a copy operation targeting `192.168.200.6`.

### Final Query (Clean)

```
source=/home/splunk/data.csv ParentImage="C:\\Users\\User067\\Downloads\\GoogleUpdate.exe"
| search CommandLine="*192.168.200.6*"
| table ParentImage CommandLine
```

---

### Findings

**Lateral Movement Command:**

| Field | Value |
|---|---|
| ParentImage | `C:\Users\User067\Downloads\GoogleUpdate.exe` |
| CommandLine | Copy operation pushing `GoogleUpdate.exe` to `\\192.168.200.6\<share>` |
| Destination Segment | Server Segment — 192.168.200.X |
| Destination Type | Shared file server accessible by other network hosts |

**Propagation Chain:**

| Stage | Action |
|---|---|
| 1 | Malware executes on User067's workstation from `C:\Users\User067\Downloads\` |
| 2 | GoogleUpdate.exe copies itself to shared drive at `192.168.200.6` via CommandLine file operation |
| 3 | Other hosts accessing the shared drive pick up the malicious binary |
| 4 | Second host infected — confirmed by two distinct `extracted_host` values in Lab 8 |

**Option Elimination Table:**

| Answer Option | Verdict | Reasoning |
|---|---|---|
| SSH connection to other hosts | **False** | No SSH connection events or authentication logs appear in the GoogleUpdate.exe command history |
| Only host the malware accessed | **False** | Disproven by the copy command targeting 192.168.200.6 and the two-host count from Lab 8 |
| Downloaded through browser on other hosts | **False** | No secondary Outlook or Chrome process chains appear on other hosts |
| Malware copied itself to file server, spreading to other hosts | **Correct** | Copy command confirmed in CommandLine table; destination confirmed as shared drive on Server Segment |

**Answer:** The malware made a copy of itself to the file server at `192.168.200.6` and expanded to other hosts on the network that access that shared drive. No direct attacker interaction with each additional host was required — the shared drive did the distribution.
