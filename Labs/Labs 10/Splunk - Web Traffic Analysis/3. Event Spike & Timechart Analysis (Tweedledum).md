# Splunk SIEM – Event Spike & Timechart Analysis

## Overview

Uniform event distribution across time is a baseline indicator of normal traffic. A sudden
spike in event volume at a single point in time is a strong anomaly signal — often indicating
a flood attack, automated burst activity, or a misconfigured tool hammering a server.

Splunk's `timechart` command is the primary tool for visualizing event volume over time.
Combined with post-spike filtering, it allows you to isolate not just when an anomaly
occurred, but who caused it.

---

## Step-by-Step Lab

### 1. Visualize Event Distribution Across Time

**Method 1 — Automatic time chart (GUI)**:

Run any broad search (e.g., `index=main`) and observe the bar chart Splunk renders
above the results table. Each bar represents event volume within a time bucket. A
single bar significantly taller than the rest indicates a spike.

**Method 2 — Explicit timechart query**:

```spl
index=main | timechart span=1h count | sort -count
```

- `timechart span=1h count` — buckets all events into 1-hour windows and counts
  events per bucket
- `sort -count` — surfaces the highest-volume time window at the top of the results

**Expected Outcome**: One time bucket stands out with a significantly higher event count
than all others. This is the anomalous spike. Note the timestamp for use in the next step.

---

### 2. Isolate the Spike Window

Filter the search to only the spike time period. Use Splunk's time picker to constrain
the search to the hour identified in Step 1, or add an explicit time filter:

```spl
index=main earliest="<spike_start>" latest="<spike_end>"
| stats count by UserAgent
| sort -count
```

Replace `<spike_start>` and `<spike_end>` with the timestamp bounds of the anomalous
hour (e.g., `earliest="2024-01-15 14:00:00"` `latest="2024-01-15 15:00:00"`).

**Expected Outcome**: One User-Agent accounts for the overwhelming majority of events
during the spike window — **Tweedledum**. All other agents contribute comparatively
negligible volume during this period.

---

### 3. Confirm the Culprit with a Split Timechart

To visually confirm which agent drove the spike, split the timechart by UserAgent:

```spl
index=main | timechart span=1h count by UserAgent
```

**Expected Outcome**: Tweedledum's line spikes sharply at the anomalous hour while
all other User-Agents remain flat. This confirms Tweedledum as the sole driver of
the event volume increase.

---

### 4. Profile Tweedledum's Activity

Examine what Tweedledum was actually doing during the spike:

```spl
index=main UserAgent="Tweedledum"
| rex field=URI "(?<Resource>[^\?]*)"
| stats count by Resource
| sort -count
```

**Expected Outcome**: A breakdown of which pages Tweedledum targeted and how many
times — revealing whether the spike represents a crawl, a flood against a specific
endpoint, or something else.

---

## Common Mistakes

1. **Using `count` without `span` in timechart**
   - `| timechart count` without a `span` argument lets Splunk auto-select the bucket
     size, which may be too coarse or too granular to surface the spike clearly.
   - Always specify `span=1h` (or appropriate granularity) for consistent results.

2. **Sorting the timechart output and stopping there**
   - `sort -count` identifies the spike window but does not reveal the cause.
   - Always follow up by filtering to that window and breaking down by UserAgent.

3. **Splitting by UserAgent too early**
   - Running `timechart count by UserAgent` across the full log before identifying
     the spike makes the chart cluttered and hard to read with many agents present.
   - Identify the spike first, then split to confirm the culprit.

4. **Assuming the spike is malicious without profiling the activity**
   - A spike could be a legitimate batch job, scheduled crawler, or misconfigured
     tool rather than an attack. Always inspect what the agent was requesting before
     drawing conclusions.
