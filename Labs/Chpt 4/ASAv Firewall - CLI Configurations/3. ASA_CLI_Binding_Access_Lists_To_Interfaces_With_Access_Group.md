# ASA_CLI_Binding_Access_Lists_To_Interfaces_With_Access_Group_Mental_Model

| Concept | Operational Meaning |
|---|---|
| `access-group` | The command that attaches an ACL to an interface in a given direction; without it the ACL is inert |
| `in interface <name>` | Applies the ACL to inbound traffic arriving on that interface; this is the standard direction for filtering traffic from a network segment into the ASA |
| One ACL per interface per direction | The ASA only allows one ACL bound inbound and one bound outbound per interface; binding a second ACL in the same direction replaces the first |
| ACL name must match exactly | The name used in `access-group` must be identical to the name used in `access-list`; a mismatch results in an empty or non-existent ACL being bound |
| Interface name vs interface ID | The name used in `access-group` is the logical name assigned to the interface (e.g. `Workstations`, `Servers`), not the physical ID (e.g. `GigabitEthernet0/1`) |
| Immediate effect | Once `access-group` is run the ACL is active immediately against live traffic; no reload or save is required for the change to take effect |
| Subsequent rule additions | After an ACL is bound, any new `access-list` entries added under that same ACL name are evaluated immediately without re-running `access-group` |
| ASDM comparison | ASDM combines `access-list` and `access-group` into a single rule-add action; the CLI exposes them as two separate required steps |

# ASA_CLI_Binding_Access_Lists_To_Interfaces_With_Access_Group_Configuration_Checklist

| Step | Task | Device | Command | Expected Result |
|---:|---|---|---|---|
| 1 | Confirm you are in global config mode | ASAv | Note prompt shows `ciscoasa(config)#` | Ready to issue access-group commands |
| 2 | Bind Workstations ACL to Workstations interface | ASAv | `access-group Workstations_access_in in interface Workstations` | Command accepted; ACL is now active on the Workstations interface |
| 3 | Bind Servers ACL to Servers interface | ASAv | `access-group Servers_access_in in interface Servers` | Command accepted; ACL is now active on the Servers interface |
| 4 | Switch to the second CMD window on Win_Station | Win_Station | Use the existing open CMD window | No new window needed |
| 5 | Ping IIS_Server again | Win_Station | `ping IIS_Server` | Replies received; ICMP is now permitted through the ASA |
| 6 | Confirm bindings are in running-config | ASAv | `show running-config access-group` | Both access-group entries appear with correct interface names |

# ASA_CLI_Binding_Access_Lists_To_Interfaces_With_Access_Group_Skeleton

ciscoasa(config)# access-group Workstations_access_in in interface Workstations
ciscoasa(config)# access-group Servers_access_in in interface Servers

! Both ACLs are now active.
! Any existing or future access-list entries under these names
! are evaluated immediately without re-running access-group.

# ASA_CLI_Binding_Access_Lists_To_Interfaces_With_Access_Group_Failure_Checks

| Symptom | Likely Cause | Fix |
|---|---|---|
| Ping still times out after running access-group | ACL name in access-group does not match the name used in access-list | Verify spelling with `show running-config access-list` and correct the access-group command |
| access-group command rejected | Interface name is wrong or does not exist on the ASA | Verify logical interface names with `show nameif` and use the exact name shown |
| Existing binding was silently replaced | A second access-group was applied inbound on the same interface | The ASA replaces the old binding; verify with `show running-config access-group` and reapply if needed |
| Hit counters stay at zero after confirmed ping | ACL is bound but the permit rule does not match the traffic | Check source, destination, and protocol in the access-list entry against actual traffic |
| Changes lost after reboot | `write memory` was not run | Run `write memory` to persist access-group bindings to startup-config |
