# Configure_Backup_Reports_Alerts_And_Recovery_Validation_Mental_Model

| Concept | Operational Meaning |
|---|---|
| Backup reporting | Long-term visibility into backup jobs, protected items, storage usage, policy adherence, restore activity, and trends |
| Backup Reports | Built-in Azure Workbook backed by Log Analytics data from vault diagnostic settings |
| Log Analytics workspace | Central destination that stores backup reporting data from Recovery Services vaults and Backup vaults |
| Vault diagnostic setting | Routing rule that sends vault diagnostic logs to Log Analytics so Backup Reports and KQL can use them |
| Recovery Services vault | Vault used by Azure Backup and Site Recovery for classic backup and disaster recovery workloads |
| Backup vault | Data Protection vault used for newer Azure Backup workloads |
| Backup item | Protected workload instance, such as Azure VM, Azure Files share, SQL database in Azure VM, disk, or blob workload |
| Backup job | Operation such as backup, restore, configure protection, disable protection, or delete backup data |
| Recovery point | Restorable backup snapshot or vault recovery point created by backup policy or on-demand backup |
| Policy adherence | Whether backup items meet expected backup schedule and retention requirements |
| Built-in Azure Monitor backup alert | Default alert generated by Azure Backup for key scenarios such as backup failure, restore failure, deletion of backup data, or soft delete changes |
| Classic backup alert | Older backup alerting model that should be phased out when using Azure Monitor based alerts |
| Action group | Notification or automation destination used for backup alerts |
| Alert processing rule | Rule that routes or suppresses backup alerts by scope, severity, condition, or schedule |
| Custom log alert | Alert built from KQL against backup diagnostic data in Log Analytics |
| Azure Resource Graph alert | Alert or query model based on near real-time Azure Resource Graph data |
| Recovery validation | Evidence-based proof that backups exist, recovery points are usable, restore jobs complete, and restored data passes checks |
| Restore test | Controlled restore to alternate location, isolated network, restored disks, or file recovery path |
| First rule | Reports require diagnostic settings. No diagnostic setting means no useful backup report history |
| Blunt rule | A green backup job does not prove recoverability. A tested restore proves recoverability |

# Configure_Backup_Reports_Alerts_And_Recovery_Validation_Diagnostic_Settings_Skeleton

```bash
# Run from Azure Cloud Shell or a workstation with Azure CLI.
# Purpose: route Recovery Services vault and Backup vault diagnostics to Log Analytics.

SUBSCRIPTION_ID="<subscription-id>"
MONITOR_RG="<monitor-resource-group>"
WORKSPACE_NAME="<workspace-name>"
RECOVERY_RG="<recovery-resource-group>"
RSV_NAME="<recovery-services-vault-name>"
BACKUP_VAULT_NAME="<backup-vault-name>"
DIAG_NAME="<diagnostic-setting-name>"

az login

az account set \
  --subscription "$SUBSCRIPTION_ID"

WORKSPACE_ID=$(az monitor log-analytics workspace show \
  --resource-group "$MONITOR_RG" \
  --workspace-name "$WORKSPACE_NAME" \
  --query id \
  --output tsv)

RSV_ID=$(az backup vault show \
  --resource-group "$RECOVERY_RG" \
  --name "$RSV_NAME" \
  --query id \
  --output tsv)

echo "Workspace ID: $WORKSPACE_ID"
echo "Recovery Services Vault ID: $RSV_ID"

az monitor diagnostic-settings categories list \
  --resource "$RSV_ID" \
  --output table

az monitor diagnostic-settings create \
  --name "$DIAG_NAME" \
  --resource "$RSV_ID" \
  --workspace "$WORKSPACE_ID" \
  --logs '[{"categoryGroup":"allLogs","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]'

az monitor diagnostic-settings show \
  --name "$DIAG_NAME" \
  --resource "$RSV_ID" \
  --output jsonc

# Backup vault diagnostics, if a Backup vault exists.
BACKUP_VAULT_ID=$(az dataprotection backup-vault show \
  --resource-group "$RECOVERY_RG" \
  --vault-name "$BACKUP_VAULT_NAME" \
  --query id \
  --output tsv 2>/dev/null)

if [ -n "$BACKUP_VAULT_ID" ]; then
  echo "Backup Vault ID: $BACKUP_VAULT_ID"

  az monitor diagnostic-settings categories list \
    --resource "$BACKUP_VAULT_ID" \
    --output table

  az monitor diagnostic-settings create \
    --name "$DIAG_NAME" \
    --resource "$BACKUP_VAULT_ID" \
    --workspace "$WORKSPACE_ID" \
    --logs '[{"categoryGroup":"allLogs","enabled":true}]' \
    --metrics '[{"category":"AllMetrics","enabled":true}]'

  az monitor diagnostic-settings show \
    --name "$DIAG_NAME" \
    --resource "$BACKUP_VAULT_ID" \
    --output jsonc
fi
```

# Configure_Backup_Reports_Alerts_And_Recovery_Validation_Action_Group_And_Processing_Rule_Skeleton

```bash
# Run from Azure Cloud Shell or a workstation with Azure CLI.
# Purpose: create an action group and alert processing rule to route backup alerts.

SUBSCRIPTION_ID="<subscription-id>"
MONITOR_RG="<monitor-resource-group>"
LOCATION="global"
ACTION_GROUP_NAME="<action-group-name>"
ACTION_GROUP_SHORT_NAME="<action-group-short-name>"
EMAIL_RECEIVER_NAME="BackupOpsEmail"
EMAIL_ADDRESS="<email-address>"
APR_NAME="<alert-processing-rule-name>"
SCOPE="/subscriptions/<subscription-id>"

az account set \
  --subscription "$SUBSCRIPTION_ID"

az monitor action-group create \
  --resource-group "$MONITOR_RG" \
  --name "$ACTION_GROUP_NAME" \
  --short-name "$ACTION_GROUP_SHORT_NAME" \
  --location "$LOCATION" \
  --action email "$EMAIL_RECEIVER_NAME" "$EMAIL_ADDRESS" usecommonalertschema

ACTION_GROUP_ID=$(az monitor action-group show \
  --resource-group "$MONITOR_RG" \
  --name "$ACTION_GROUP_NAME" \
  --query id \
  --output tsv)

cat > backup-alert-processing-rule.json <<EOF
{
  "location": "$LOCATION",
  "properties": {
    "enabled": true,
    "scopes": [
      "$SCOPE"
    ],
    "conditions": [
      {
        "field": "MonitorService",
        "operator": "Equals",
        "values": [
          "Azure Backup"
        ]
      }
    ],
    "actions": [
      {
        "actionType": "AddActionGroups",
        "actionGroupIds": [
          "$ACTION_GROUP_ID"
        ]
      }
    ],
    "description": "Route Azure Backup alerts to Backup Operations action group"
  }
}
EOF

az resource create \
  --resource-group "$MONITOR_RG" \
  --name "$APR_NAME" \
  --resource-type "Microsoft.AlertsManagement/actionRules" \
  --api-version "2023-05-01-preview" \
  --is-full-object \
  --properties @backup-alert-processing-rule.json

az monitor action-group show \
  --resource-group "$MONITOR_RG" \
  --name "$ACTION_GROUP_NAME" \
  --output jsonc

az resource show \
  --resource-group "$MONITOR_RG" \
  --name "$APR_NAME" \
  --resource-type "Microsoft.AlertsManagement/actionRules" \
  --api-version "2023-05-01-preview" \
  --output jsonc
```

# Configure_Backup_Reports_Alerts_And_Recovery_Validation_Custom_Log_Alert_Skeleton

```bash
# Run from Azure Cloud Shell or a workstation with Azure CLI.
# Purpose: create a custom scheduled query alert for backup or restore failures in Log Analytics.
# Validate the KQL query manually before enabling alerting.

SUBSCRIPTION_ID="<subscription-id>"
MONITOR_RG="<monitor-resource-group>"
LOCATION="<location>"
WORKSPACE_RESOURCE_ID="<workspace-resource-id>"
ACTION_GROUP_NAME="<action-group-name>"
LOG_ALERT_NAME="la-backup-restore-failure"
SEVERITY="1"
WINDOW_SIZE="15m"
EVALUATION_FREQUENCY="15m"

az account set \
  --subscription "$SUBSCRIPTION_ID"

ACTION_GROUP_ID=$(az monitor action-group show \
  --resource-group "$MONITOR_RG" \
  --name "$ACTION_GROUP_NAME" \
  --query id \
  --output tsv)

az monitor scheduled-query create \
  --resource-group "$MONITOR_RG" \
  --name "$LOG_ALERT_NAME" \
  --location "$LOCATION" \
  --scopes "$WORKSPACE_RESOURCE_ID" \
  --condition "count 'BackupFailureQuery' > 0 at least 1 violations out of 1 aggregated points" \
  --condition-query BackupFailureQuery="AzureDiagnostics | where TimeGenerated > ago(30m) | where ResourceProvider =~ 'MICROSOFT.RECOVERYSERVICES' | where ResultType has_any ('Failed','Failure','Error') or Level in ('Error','Critical')" \
  --window-size "$WINDOW_SIZE" \
  --evaluation-frequency "$EVALUATION_FREQUENCY" \
  --severity "$SEVERITY" \
  --action-groups "$ACTION_GROUP_ID" \
  --description "Custom alert for Recovery Services backup or restore failures"

az monitor scheduled-query show \
  --resource-group "$MONITOR_RG" \
  --name "$LOG_ALERT_NAME" \
  --output jsonc
```

# Configure_Backup_Reports_Alerts_And_Recovery_Validation_KQL_Query_Skeleton

```kusto
// Purpose: validate backup reports, alerts, job history, restore activity, and vault operations.
// Run in the Log Analytics workspace where Recovery Services vault and Backup vault diagnostics are routed.
// Table names vary by diagnostic schema and resource-specific settings.
// Start with broad discovery, then narrow to Recovery Services and backup operations.

//
// 1. Confirm Recovery Services diagnostic data exists.
//
AzureDiagnostics
| where TimeGenerated > ago(30d)
| where ResourceProvider =~ "MICROSOFT.RECOVERYSERVICES"
| summarize Count=count(), LastRecord=max(TimeGenerated) by Resource, Category, OperationName, ResultType
| order by LastRecord desc

//
// 2. Confirm Data Protection backup vault diagnostic data exists if Backup vault is used.
//
AzureDiagnostics
| where TimeGenerated > ago(30d)
| where ResourceProvider =~ "MICROSOFT.DATAPROTECTION"
| summarize Count=count(), LastRecord=max(TimeGenerated) by Resource, Category, OperationName, ResultType
| order by LastRecord desc

//
// 3. Backup and restore failures from vault diagnostic logs.
//
AzureDiagnostics
| where TimeGenerated > ago(30d)
```

# Configure_Backup_Reports_Alerts_And_Recovery_Validation_Rollback

| Change | Rollback Command / Action | Risk |
|---|---|---|
| Diagnostic setting routed to wrong workspace | Delete diagnostic setting and recreate with correct workspace | Report data history remains split across workspaces |
| Diagnostic setting too broad | Update categories or destination | Reduces report and query visibility |
| Workspace retention too high | Lower workspace retention setting | Older report data may be removed after retention enforcement |
| Workspace retention too low | Increase retention setting | Does not restore data already purged |
| Classic alerts disabled too early | Re-enable classic alerts temporarily if migration was not ready | Duplicate alerting risk if Azure Monitor alerts remain active |
| Built-in job failure alerts disabled by mistake | Re-enable alerts for job failures | Backup and restore failures may not notify operators while disabled |
| Action group created with wrong receiver | Update or recreate action group | Wrong team may receive alerts |
| Alert processing rule routes too much | Narrow scope or conditions | Alert noise can increase |
| Suppression rule hides real backup failures | Disable or delete suppression rule | Critical backup failures may be missed |
| Custom log alert too noisy | Disable or refine scheduled query rule | Excess notifications |
| Backup report workbook customized poorly | Delete copied workbook or revert saved version | Removes custom view only |
| Restore test resources left behind | Delete test VM, disks, NICs, public IPs, and test resource group if safe | Cost and naming confusion |
| Evidence folder created locally | Delete evidence folder | Deletes local evidence only |

# Configure_Backup_Reports_Alerts_And_Recovery_Validation_Failure_Checks

| Symptom | Likely Layer | First Check | Do Not Start With |
|---|---|---|---|
| Backup Reports show no data | Vault diagnostics missing, wrong workspace, or ingestion delay | Diagnostic settings and workspace scope | Recreating vault |
| Backup Reports missing current day | Report convention excludes current partial day | Check completed UTC days | Changing retention |
| Reports show wrong vaults | Portal filters wrong | Subscription, vault, location, and workload filters | Editing diagnostic settings |
| Report data delayed | Initial diagnostic push or workbook delay | Wait expected ingestion window and query raw logs | Rebuilding workspace |
| KQL shows no Recovery Services logs | No diagnostic setting or wrong table mode | `az monitor diagnostic-settings list` | Recreating backup jobs |
| Built-in alerts not visible | Monitoring setting disabled or no fired alert | Vault Properties > Monitoring Settings | Creating duplicate alerts |
| Email not received | Action group receiver or alert processing rule issue | Test action group and check processing rule | Disabling backup alerts |
| Duplicate backup alerts | Classic and Azure Monitor alerts both active | Check classic alert opt-out state | Deleting action group |
| Backup failure alert not routed | Processing rule scope or condition mismatch | Check monitor service, scope, and severity | Recreating vault |
| Suppression hides backup failure | Suppression processing rule matched | Review alert processing rules | Assuming job succeeded |
| Custom log alert never fires | KQL query returns no rows or wrong workspace | Run query manually | Editing action group first |
| Custom log alert fires constantly | KQL query too broad | Add categories, result filters, and time windows | Disabling all alerts |
| Recovery validation fails at protected item lookup | Wrong workload type, vault, or item name | List containers and backup items | Re-running backup immediately |
| Recovery point missing | Backup not completed or wrong protected item selected | Check job list and item name | Running restore blindly |
| Restore test job fails | Target RG, storage, permissions, region, or workload issue | Backup job error details | Deleting recovery points |
| Restored data is stale | RPO target not met | Compare recovery point timestamp to incident assumption | Marking validation pass |
| Restored VM boots but app fails | Application dependency, identity, DNS, firewall, or service startup issue | App owner validation checklist | Blaming Azure Backup only |
| Restore test resources remain | Cleanup step missed | List restore test resource group | Leaving resources for later |
| RTO cannot be proven | Timestamps not captured | Evidence register and job details | Estimating after the fact |
| Policy adherence report shows drift | Item assigned to wrong policy or retention mismatch | Policy report and backup item details | Editing all policies at once |
