# Configure_Alerts_Action_Groups_Alert_Processing_Rules_And_Workbook_Views_Mental_Model

| Concept | Operational Meaning |
|---|---|
| Alert rule | Definition that watches a resource, signal, and condition, then fires an alert when the condition is met |
| Alert instance | Fired alert generated by an alert rule or Azure service |
| Signal | Metric, log query, activity log event, resource health event, or service health event used to trigger an alert |
| Scope | Resource, resource group, workspace, or subscription boundary monitored by the alert rule |
| Condition | Threshold, query result, event match, or dynamic condition that determines when an alert fires |
| Evaluation frequency | How often Azure Monitor checks the alert rule condition |
| Window size | Lookback period used by the alert rule when evaluating metrics or logs |
| Severity | Alert importance from Sev0 critical through Sev4 verbose |
| Action group | Reusable notification and automation target used by alert rules or alert processing rules |
| Notification receiver | Email, SMS, voice, or Azure app push destination |
| Automation receiver | Webhook, secure webhook, Logic App, Azure Function, Automation runbook, Event Hub, or ITSM action |
| Common alert schema | Standardized payload format for alert notifications |
| Alert processing rule | Rule that modifies fired alerts by applying action groups or suppressing notifications |
| Suppression rule | Alert processing rule that prevents action groups from running during maintenance or filtered conditions |
| Apply action group rule | Alert processing rule that adds one or more action groups to matching fired alerts |
| Workbook view | Operator-facing visual report that combines KQL, metrics, parameters, text, and Resource Graph views |
| First rule | Build the action group before attaching alerts to it |
| Blunt rule | An alert rule without a tested action group is only half-built |

# Configure_Alerts_Action_Groups_Alert_Processing_Rules_And_Workbook_Views_Action_Group_Skeleton

```bash
# Run from Azure Cloud Shell or a workstation with Azure CLI.
# Purpose: create a reusable action group for Azure Monitor alerts.

SUBSCRIPTION_ID="<subscription-id>"
MONITOR_RG="<monitor-resource-group>"
LOCATION="global"
ACTION_GROUP_NAME="<action-group-name>"
ACTION_GROUP_SHORT_NAME="<short-name>"
EMAIL_RECEIVER_NAME="PrimaryOnCall"
EMAIL_ADDRESS="<email-address>"
WEBHOOK_RECEIVER_NAME="PrimaryWebhook"
WEBHOOK_URI="<webhook-uri>"

az account set \
  --subscription "$SUBSCRIPTION_ID"

az monitor action-group create \
  --resource-group "$MONITOR_RG" \
  --name "$ACTION_GROUP_NAME" \
  --short-name "$ACTION_GROUP_SHORT_NAME" \
  --location "$LOCATION" \
  --action email "$EMAIL_RECEIVER_NAME" "$EMAIL_ADDRESS" usecommonalertschema

# Optional webhook receiver.
# Use only when the webhook endpoint is approved and can process Azure Monitor alert payloads.

# az monitor action-group update \
#   --resource-group "$MONITOR_RG" \
#   --name "$ACTION_GROUP_NAME" \
#   --add-action webhook "$WEBHOOK_RECEIVER_NAME" "$WEBHOOK_URI" usecommonalertschema

ACTION_GROUP_ID=$(az monitor action-group show \
  --resource-group "$MONITOR_RG" \
  --name "$ACTION_GROUP_NAME" \
  --query id \
  --output tsv)

echo "$ACTION_GROUP_ID"
```

# Configure_Alerts_Action_Groups_Alert_Processing_Rules_And_Workbook_Views_Metric_Alert_Skeleton

```bash
# Run from Azure Cloud Shell or a workstation with Azure CLI.
# Purpose: create a metric alert rule tied to the reusable action group.

SUBSCRIPTION_ID="<subscription-id>"
MONITOR_RG="<monitor-resource-group>"
ACTION_GROUP_NAME="<action-group-name>"
TARGET_RESOURCE_ID="<target-resource-id>"
METRIC_ALERT_NAME="<metric-alert-name>"
METRIC_NAME="Percentage CPU"
THRESHOLD="80"
SEVERITY="2"
WINDOW_SIZE="5m"
EVALUATION_FREQUENCY="1m"

az account set \
  --subscription "$SUBSCRIPTION_ID"

ACTION_GROUP_ID=$(az monitor action-group show \
  --resource-group "$MONITOR_RG" \
  --name "$ACTION_GROUP_NAME" \
  --query id \
  --output tsv)

az monitor metrics alert create \
  --resource-group "$MONITOR_RG" \
  --name "$METRIC_ALERT_NAME" \
  --scopes "$TARGET_RESOURCE_ID" \
  --condition "avg $METRIC_NAME > $THRESHOLD" \
  --window-size "$WINDOW_SIZE" \
  --evaluation-frequency "$EVALUATION_FREQUENCY" \
  --severity "$SEVERITY" \
  --description "Metric alert for $METRIC_NAME greater than $THRESHOLD" \
  --action "$ACTION_GROUP_ID"

az monitor metrics alert show \
  --resource-group "$MONITOR_RG" \
  --name "$METRIC_ALERT_NAME" \
  --output jsonc
```

```bash
# Purpose: create a scheduled query alert from a validated KQL query.
# The scheduled-query CLI extension installs automatically when first used on supported Azure CLI versions.

SUBSCRIPTION_ID="<subscription-id>"
MONITOR_RG="<monitor-resource-group>"
LOCATION="<location>"
ACTION_GROUP_NAME="<action-group-name>"

WORKSPACE_RESOURCE_ID="<workspace-resource-id>"
SEVERITY="2"
WINDOW_SIZE="5m"
EVALUATION_FREQUENCY="5m"

az account set \
  --subscription "$SUBSCRIPTION_ID"

ACTION_GROUP_ID=$(az monitor action-group show \
  --resource-group "$MONITOR_RG" \
  --name "$ACTION_GROUP_NAME" \
  --query id \
  --output tsv)

# Replace Placeholder_1 query with a table that exists in your workspace.
# For Azure VM Windows event logs, Event must exist.
# For AzureDiagnostics, diagnostic settings must already route logs to the workspace.

az monitor scheduled-query create \
  --resource-group "$MONITOR_RG" \
  --name "$LOG_ALERT_NAME" \
  --location "$LOCATION" \
  --scopes "$WORKSPACE_RESOURCE_ID" \
  --condition "count 'Placeholder_1' > 0 at least 1 violations out of 1 aggregated points" \
  --condition-query Placeholder_1="AzureDiagnostics | where TimeGenerated > ago(5m) | where Level in ('Error','Critical')" \
  --window-size "$WINDOW_SIZE" \
  --evaluation-frequency "$EVALUATION_FREQUENCY" \
  --severity "$SEVERITY" \
  --action-groups "$ACTION_GROUP_ID" \
  --description "Log alert for recent error or critical AzureDiagnostics records"

az monitor scheduled-query show \
  --resource-group "$MONITOR_RG" \
  --name "$LOG_ALERT_NAME" \
  --output jsonc
```

# Configure_Alerts_Action_Groups_Alert_Processing_Rules_And_Workbook_Views_Alert_Processing_Rule_Skeleton

```bash
# Run from Azure Cloud Shell or a workstation with Azure CLI.
```
```bash
# Purpose: create an alert processing rule that applies an action group to fired alerts.
# Uses generic ARM deployment because alert processing rules are Microsoft.AlertsManagement/actionRules resources.

SUBSCRIPTION_ID="<subscription-id>"
MONITOR_RG="<monitor-resource-group>"
LOCATION="global"
ACTION_GROUP_NAME="<action-group-name>"
APR_NAME="<alert-processing-rule-name>"
SCOPE="/subscriptions/<subscription-id>"

az account set \
  --subscription "$SUBSCRIPTION_ID"

ACTION_GROUP_ID=$(az monitor action-group show \
  --resource-group "$MONITOR_RG" \
  --name "$ACTION_GROUP_NAME" \
  --query id \
  --output tsv)

cat > apr-apply-action-group.json <<EOF
{
  "location": "$LOCATION",
  "properties": {
    "enabled": true,
    "scopes": [
      "$SCOPE"
    ],
    "conditions": [
      {
        "field": "Severity",
        "operator": "Equals",
        "values": [
          "Sev0",
          "Sev1",
          "Sev2"
        ]
      }
    ],
    "actions": [
      {
        "actionType": "AddActionGroups",
        "actionGroupIds": [
          "$ACTION_GROUP_ID"
        ]
      }
    ],
    "description": "Apply action group to high and medium severity alerts"
  }
}
EOF

az resource create \
  --resource-group "$MONITOR_RG" \
  --name "$APR_NAME" \
  --resource-type "Microsoft.AlertsManagement/actionRules" \
  --api-version "2023-05-01-preview" \
  --is-full-object \
  --properties @apr-apply-action-group.json

az resource show \
  --resource-group "$MONITOR_RG" \
  --name "$APR_NAME" \
  --resource-type "Microsoft.AlertsManagement/actionRules" \
  --api-version "2023-05-01-preview" \
  --output jsonc
```

# Configure_Alerts_Action_Groups_Alert_Processing_Rules_And_Workbook_Views_Rollback

| Change | Rollback Command / Action | Risk |
|---|---|---|
| Action group created with wrong receiver | `az monitor action-group delete --resource-group "<monitor-resource-group>" --name "<action-group-name>"` then recreate | Deletes reusable notification target |
| Email receiver wrong | Update or recreate action group with correct email receiver | Alerts may notify wrong recipient until fixed |
| Webhook receiver wrong | Remove webhook action or recreate action group | Alerts may call wrong endpoint |
| Metric alert too noisy | `az monitor metrics alert update --resource-group "<monitor-resource-group>" --name "<metric-alert-name>" --enabled false` | Disables that alert |
| Metric threshold wrong | Update alert threshold or delete and recreate alert | Can create missed alerts or noise |
| Log alert query wrong | Update scheduled query rule after validating KQL | Bad KQL can generate false positives or no alerts |
| Activity log alert too broad | Narrow scope or condition | Broad activity alerts can create noise |
| Alert processing rule applied wrong action group | Delete or update `Microsoft.AlertsManagement/actionRules` resource | Can route alerts to wrong team |
| Suppression rule too broad | Disable or delete suppression rule immediately | Can hide real incidents |
| Workbook saved in wrong resource group | Delete workbook resource and save again in correct group | Deletes workbook view only |
| Evidence files created locally | Delete evidence folder | Removes local evidence only |

# Configure_Alerts_Action_Groups_Alert_Processing_Rules_And_Workbook_Views_Failure_Checks

| Symptom | Likely Layer | First Check | Do Not Start With |
|---|---|---|---|
| Alert rule exists but nobody is notified | Missing or disabled action group | Check action group ID attached to alert | Recreating target resource |
| Action group test email not received | Email verification, spam filtering, receiver typo | Test action group from portal | Changing alert threshold |
| Metric alert never fires | Wrong metric name, wrong aggregation, threshold too high | `az monitor metrics list-definitions` and metric chart | Recreating action group |
| Metric alert fires too often | Threshold too low or stateless behavior | Review window size, evaluation frequency, and threshold | Disabling all alerts |
| Log alert never fires | Query returns no rows or wrong workspace scope | Run KQL manually in Log Analytics | Editing action group |
| Log alert fires constantly | Query too broad | Add filters and summarize logic | Deleting workspace |
| Scheduled query create fails | CLI extension issue or invalid condition syntax | Confirm Azure CLI version and scheduled-query help | Switching to portal without recording syntax |
| Activity log alert does not fire | Wrong operation name, scope, category, or status | Inspect AzureActivity records first | Changing diagnostic settings |
| Alert processing rule does not apply | Scope, filters, or propagation delay | Check rule scope and wait up to expected propagation window | Recreating alert rules |
| Suppression rule hides too much | Scope or filter too broad | Disable suppression rule | Deleting action group |
| Alerts show fired but action groups suppressed | Suppression alert processing rule matched | Review processing rules | Recreating alert rule |
| Workbook shows no alerts | Wrong data source or query scope | Check Azure Resource Graph query and tenant/subscription scope | Rebuilding workbook |
| Workbook cannot be saved | RBAC issue | Confirm Monitoring Contributor or workbook write permission | Changing queries |
| Duplicate notifications | Multiple alert rules or multiple action groups | Inventory alert rules and processing rules | Disabling email globally |
| Wrong team receives alerts | Reused action group attached to wrong alert | Check alert rule actions and processing rules | Deleting all alert rules |
| Common schema payload unexpected | Receiver does not parse common alert schema | Review receiver payload expectations | Turning off alerting |
| Action group region issue | Service Health or regional processing mismatch | Confirm action group processing region | Recreating all alerts |
| Alert closed but issue still exists | User response changed manually | Check monitor condition versus user response | Assuming incident is resolved |
