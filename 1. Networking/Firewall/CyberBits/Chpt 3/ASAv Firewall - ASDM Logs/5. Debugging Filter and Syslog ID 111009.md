# ASDM_Debugging_Filter_And_Syslog_111009_Mental_Model
| Concept | Operational Meaning |
|---|---|
| Default filter severity | ASDM logging defaults to Informational (severity 1-6); severity 7 messages are suppressed unless explicitly enabled |
| Debugging severity (7) | The lowest priority severity level; captures all events including routine command executions that produce no config change |
| Syslog ID 111009 | %ASA-7-111009; fires when a user runs a command on the ASA that does not alter the running configuration (read-only commands such as `show` commands) |
| 111009 vs config change logs | 111009 records non-modifying commands; a different message ID fires when an actual configuration change is made |
| "Filter on severity" mode | ASDM logging filter mode that passes all messages at or above the selected severity level |
| Debugging as the filter floor | Setting the filter to Debugging (7) passes all severity levels 0-7; nothing is suppressed |
| Clear Content before filter change | Best practice to clear the panel before changing the filter so old entries under a different filter do not mix with new entries |
| First two 111009 entries after filter change | The Apply action itself triggers ASDM to issue internal show/read commands, which the ASA logs as 111009 events |

# ASDM_Debugging_Filter_And_Syslog_111009_Clear_And_Filter_Skeleton
Step 1 - Clear the panel:
  Home screen
  -> Latest ASDM Syslog Messages panel
  -> Right-click any log entry
  -> Select: "Clear Content"

Step 2 - Change filter to Debugging:
  Home screen
  -> Latest ASDM Syslog Messages panel
  -> Click: gear icon ("Configure ASDM Syslog Filters")
  -> Double-click: "ASDM" logging entry
  -> Field: "Filter on severity"
  -> Change to: Debugging
  -> Click: Ok
  -> Click: Apply

Equivalent CLI command issued by ASDM:
  logging asdm debugging

# ASDM_Debugging_Filter_And_Syslog_111009_Verification_Skeleton
ASDM GUI Path:
  Home screen
  -> Latest ASDM Syslog Messages panel
  -> Locate entries with severity 7 and Syslog ID 111009
  -> Hover over entry

Expected syslog message:
  ID:          111009
  Severity:    7 (Debugging)
  Description: "The user ran a command, but no actual changes were made
                to the configuration."

Expected log format example:
  %ASA-7-111009: User 'admin' executed cmd: show ...

Why two 111009 entries appear first:
  ASDM issues internal read commands as part of the Apply action;
  the ASA logs each one as a separate 111009 event since no config
  was modified by those reads.

CLI verification:
  show logging asdm
  show running-config | include logging asdm

# ASDM_Debugging_Filter_And_Syslog_111009_Failure_Checks
| Symptom | What to Check | What Usually Broke |
|---|---|---|
| Severity 7 entries do not appear after changing filter | Confirm Apply was clicked after Ok | Filter change was staged but not applied; click Apply |
| 111009 entries do not appear | Confirm filter is set to Debugging, not Informational | Filter is still at Informational which suppresses severity 7 |
| Panel is empty after filter change | Panel was cleared but no new events have been generated | Interact with ASDM (click Apply or navigate) to trigger new 111009 events |
| Hover tooltip does not show the expected description | Cursor is not positioned directly over the log row text | Reposition cursor directly over the 111009 entry row |
| Filter dropdown does not include Debugging option | ASDM version or display issue | Confirm ASDM version supports full severity range; try scrolling the dropdown |
| More than two 111009 entries appear | Expected; any read-only command issued since the filter change was applied generates a new 111009 | Normal behavior; the question targets only the first two entries |
| `logging asdm debugging` does not appear in running config after Apply | Config was not saved or Apply did not execute cleanly | Re-apply the filter and verify with `show running-config \| include logging asdm` |
