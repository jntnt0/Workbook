# ASDM_Event_List_Filter_Syslog_106023_Mental_Model
| Concept | Operational Meaning |
|---|---|
| Event list filter mode | An alternative to severity-based filtering; locks the ASDM syslog panel to only display messages matching a specific list of syslog IDs |
| Message ID filter | A rule within an event list that specifies one or more syslog message IDs to include; all other IDs are suppressed |
| Syslog ID 106023 | %ASA-4-106023; generated when the ASA denies a packet based on an ACL rule; severity is always 4 (Warning) |
| Severity 4 (Warning) | The ASA treats ACL-denied traffic as a warning; it is not informational because the traffic was actively blocked by policy |
| Event list vs severity filter | Severity filter passes all messages at or above a threshold; event list passes only the exact IDs specified regardless of severity |
| Test-NetConnection port 80 | PowerShell tool that attempts a TCP connection to the specified host and port; a failed TCP test to port 80 on IIS_Server triggers an ACL deny on the ASA |
| Single event type in output | Because the filter is locked to 106023, every entry in the panel after running the test will be the same message ID, severity, and description |
| `Deny tcp src Workstations dst Servers` | The description pattern in 106023 entries; identifies the denying interface pair and protocol |

# ASDM_Event_List_Filter_Syslog_106023_Generate_Traffic_Skeleton
Step 4 - Generate denied TCP traffic:
  Open PowerShell on Windows workstation
  Run: Test-NetConnection -ComputerName iis_server -Port 80

Expected PowerShell output:
  TcpTestSucceeded : False
  PingSucceeded    : True

Expected ASA behavior:
  ICMP (ping) is permitted; no 106023 generated for ICMP
  TCP port 80 is denied by ACL; ASA generates %ASA-4-106023
  Only the 106023 entry appears in the panel due to the active filter

# ASDM_Event_List_Filter_Syslog_106023_Verification_Skeleton
ASDM GUI Path:
  Home screen
  -> Latest ASDM Syslog Messages panel
  -> Confirm all visible entries have severity 4 and ID 106023

Expected syslog message:
  ID:          106023
  Severity:    4 (Warning)
  Description: "Deny tcp src Workstations:<workstation-ip>/<port>
                dst Servers:<iis-ip>/80 by access-group"

Expected panel behavior:
  No ICMP entries appear (ICMP is permitted but 106023 is TCP-deny only)
  No severity 6 or 7 entries appear (event list suppresses all other IDs)
  Every entry in the panel is the same message ID, severity, and description pattern

CLI verification:
  show logging asdm
  show running-config | include logging list | logging asdm
  show access-list | include deny

# ASDM_Event_List_Filter_Syslog_106023_Failure_Checks
| Symptom | What to Check | What Usually Broke |
|---|---|---|
| Panel shows no entries after Test-NetConnection | Confirm the event list filter is applied and TCP port 80 is actually being denied | Filter not applied (Apply not clicked) or port 80 is permitted by an access rule |
| Panel shows entries other than 106023 | Confirm mode is set to "Use event list" not "Filter on severity" | Mode was not changed from severity-based filtering; event list is not active |
| 106023 entries do not appear but TCP test fails | Confirm logging is enabled for deny actions on the relevant ACL | ACL is configured without the `log` keyword; denied traffic is not generating syslog output |
| Event list "Only106023" is not available in the dropdown | Confirm the New event list was saved before clicking Apply | Event list creation dialog was closed without saving; recreate the event list |
| PowerShell command returns TcpTestSucceeded : True | Port 80 is permitted by the ASA access rules | Expected if an access rule allowing port 80 is present; check access rules and remove or adjust if the lab expects a deny |
| All entries show severity 6 instead of 4 | A different syslog ID is appearing in the panel | Event list is not filtering correctly; confirm Message ID field contains exactly `106023` with no extra characters |
| Apply button is greyed out | No change was detected by ASDM | Make a minor change in the dialog (toggle and toggle back) to force the Apply button to activate |
| `logging list Only106023` does not appear in running config | Apply was not completed or ASDM lost the session | Re-create the event list and apply again; verify with `show running-config \| include logging list` |
