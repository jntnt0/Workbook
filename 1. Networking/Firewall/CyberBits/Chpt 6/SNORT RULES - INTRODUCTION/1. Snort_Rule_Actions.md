# Snort_Rule_Actions_Mental_Model

| Concept | Operational Meaning |
|---|---|
| Rule action | The first field in every Snort rule header; determines what Snort does when a packet matches |
| alert | Generates an alert and logs the matching packet |
| log | Logs the matching packet only; no alert is generated |
| pass | Ignores the matching packet; no alert and no log |
| drop | Inline mode only; blocks the packet and logs it |
| reject | Inline mode only; blocks the packet, logs it, and sends a TCP reset if TCP or ICMP port unreachable if UDP |
| sdrop | Inline mode only; blocks the packet silently; no log entry is created |
| Inline mode requirement | drop, reject, and sdrop are only available when Snort is running in inline (IPS) mode, not passive IDS mode |
| Default vs inline actions | alert, log, and pass are the three default actions available in all Snort modes |
| Rule header position | The action is always the leftmost element of the rule header, before protocol, IP, and port |
| Choosing drop vs reject | Use drop when silent blocking is acceptable; use reject when you want the sender notified via reset or unreachable |
| Choosing drop vs sdrop | Both block; drop logs the event, sdrop does not; use sdrop only when log volume is a concern |

# Snort_Rule_Actions_Configuration_Checklist

| Step | Task | Device | Command | Expected Result |
|---:|---|---|---|---|
| 1 | Navigate to the Snort local rules file | Ubuntu Workstation | sudo nano /etc/snort/rules/local.rules | Rules file opens for editing |
| 2 | Write a rule using the target action keyword as the first field | Ubuntu Workstation | <action> <protocol> <src_ip> <src_port> -> <dst_ip> <dst_port> (msg:"<label>"; sid:<id>;) | Rule is present in the file with correct syntax |
| 3 | Save the rules file | Ubuntu Workstation | Ctrl+O, Enter, Ctrl+X | File is saved and editor exits |
| 4 | Test Snort configuration syntax before running | Ubuntu Workstation | sudo snort -T -c /etc/snort/snort.conf | Snort reports configuration OK with no errors |
| 5 | Run Snort in IDS mode against a target interface | Ubuntu Workstation | sudo snort -A console -q -c /etc/snort/snort.conf -i <interface> | Snort starts and begins monitoring traffic |
| 6 | Run Snort in inline IPS mode (required for drop/reject/sdrop) | Ubuntu Workstation | sudo snort -Q --daq nfq --daq-var queue=0 -c /etc/snort/snort.conf | Snort starts in inline mode; drop/reject/sdrop actions are active |
| 7 | Generate test traffic to trigger the rule | Win_Station or Ubuntu Workstation | ping <target_ip> or browser request | Traffic matching the rule is produced |
| 8 | Observe console output for alert or log events | Ubuntu Workstation | Review terminal output | Alert message with rule msg field appears if action is alert |
| 9 | Verify no alert fires when action is pass | Ubuntu Workstation | Review terminal output | No output for matching traffic when rule action is pass |
| 10 | Verify blocked traffic when action is drop | Win_Station | ping <target_ip> | Ping requests time out; no response received |
| 11 | Verify TCP reset is sent when action is reject on TCP | Win_Station | Attempt TCP connection | Connection is immediately refused rather than timing out |

# Snort_Rule_Actions_Skeleton

Alert Skeleton
alert <protocol> <src_ip> <src_port> -> <dst_ip> <dst_port> (msg:"<ALERT_LABEL>"; sid:<SID>;)

Log Skeleton
log <protocol> <src_ip> <src_port> -> <dst_ip> <dst_port> (msg:"<LOG_LABEL>"; sid:<SID>;)

Pass Skeleton
pass <protocol> <src_ip> <src_port> -> <dst_ip> <dst_port> (msg:"<PASS_LABEL>"; sid:<SID>;)

Drop Skeleton (inline mode required)
drop <protocol> <src_ip> <src_port> -> <dst_ip> <dst_port> (msg:"<DROP_LABEL>"; sid:<SID>;)

Reject Skeleton (inline mode required)
reject <protocol> <src_ip> <src_port> -> <dst_ip> <dst_port> (msg:"<REJECT_LABEL>"; sid:<SID>;)

sdrop Skeleton (inline mode required)
sdrop <protocol> <src_ip> <src_port> -> <dst_ip> <dst_port> (msg:"<SDROP_LABEL>"; sid:<SID>;)

ICMP Lab Example
drop icmp any any -> any any (msg:"ICMP PACKET DETECTED"; sid:1000001;)

| 2 | Check alert log for triggered rules | Ubuntu Workstation | cat /var/log/snort/alert | Alert entries appear with matching msg field |

| Step | Task | Device | Command | Expected Result |
|---:|---|---|---|---|
| 1 | Open the local rules file | Ubuntu Workstation | sudo nano /etc/snort/rules/local.rules | File opens for editing |
| 2 | Comment out or delete the target rule | Ubuntu Workstation | Prefix the rule line with # or delete the line | Rule is disabled without removing the file |

# Snort_Rule_Actions_Failure_Checks

| Symptom | Command | What Usually Broke |
|---|---|---|
| drop or reject rule has no effect on traffic | sudo snort -V and check mode | Snort is running in passive IDS mode, not inline; drop/reject/sdrop require inline mode |
| Snort throws an error on startup | sudo snort -T -c /etc/snort/snort.conf | Rule syntax error; check action spelling, missing semicolons, or malformed rule options |
| Alert fires but traffic is not blocked | Review rule action field | Action is alert or log, not drop; change action to drop and restart in inline mode |
| No output at all when traffic is generated | ps aux | grep snort | Snort is not running or is listening on the wrong interface |
| sdrop rule blocks traffic but no log entry appears | cat /var/log/snort/alert | Expected behavior; sdrop intentionally suppresses logging |
| reject sends no TCP reset | Verify protocol field in rule | Protocol is not TCP; reject only sends reset for TCP and port unreachable for UDP |
| Rule file changes have no effect | sudo systemctl status snort | Snort was not restarted after editing the rules file |
