# Splunk SIEM – Attack Identification

## Overview

Web server logs capture not just normal traffic but also attack attempts. Four distinct
attack types are identifiable through URI and User-Agent field analysis in Splunk:
**Buffer Overflow**, **Null Byte Injection**, and **SQL Injection** (two variants).
Each has a distinct signature that can be detected through field length analysis,
character pattern matching, and URL decoding.

---

## Step-by-Step Lab

### 1. Buffer Overflow Detection

A buffer overflow attack attempts to exceed the memory boundary of an input field by
submitting abnormally long strings. In web logs, this manifests as unusually long URI
parameters.

**Step 1 — Broad length filter**:

```spl
index=main | where length(URI)>100
```

Surfaces all events where the full URI exceeds 100 characters — a reasonable starting
threshold for anomalous input length.

**Step 2 — Isolate long parameter values specifically**:

```spl
index=main
| rex field=URI max_match=0 "\=(?<Parameter>[^&]{100,})"
| search Parameter=*
```

**Regex breakdown**:
- `\=` — matches the `=` sign that precedes a parameter value
- `[^&]{100,}` — matches 100 or more characters that are not `&` (the parameter
  delimiter), capturing the full value of any oversized parameter
- `max_match=0` — extracts all matching instances per event, not just the first

**Expected Outcome**: One Wonderlander appears across multiple events with
progressively varying parameter lengths — three failed attempts followed by a
successful attempt at a different length, indicating iterative probing to find
the overflow boundary.

---

### 2. Null Byte Injection Detection

A null byte (`\0`) signals string termination in many programming languages. Injecting
one into a URL parameter can cause the application to ignore everything after it —
bypassing input validation, file extension checks, or access controls.

In URL encoding, null byte = `%00`. It has no legitimate purpose in a URL.

**Detection query**:

```spl
index=main URI=*%00*
```

**Expected Outcome**: All events containing `%00` in the URI are returned. A single
Wonderlander is responsible for all null byte injection attempts, making attribution
straightforward.

**To decode and inspect the full payload**:

```spl
index=main URI=*%00*
| eval Decoded=urldecode(URI)
```

**Expected Outcome**: The `Decoded` field renders the null byte injection payload in
plain text, revealing what the attacker was attempting to access or bypass.

---

### 3. SQL Injection Detection — Boolean Method (HumptyDumpty)

SQL injection manipulates database queries by injecting SQL syntax into user-supplied
input. HumptyDumpty used the **OR boolean expression** method — injecting a condition
that always evaluates to true to bypass authentication or access controls.

Example payload: `' OR '1'='1`

Because these characters are URL-encoded in transit, raw URI inspection may not reveal
the attack clearly. Use Splunk's `urldecode` function to render the payload in plain text.

**Step 1 — Search HumptyDumpty's activity**:

```spl
index=main UserAgent="HumptyDumpty"
```

**Step 2 — Decode URI parameters**:

```spl
index=main UserAgent="HumptyDumpty"
| eval Decoded=urldecode(URI)
```

**Expected Outcome**: Two injection attempts are visible. The first attempt fails
(malformed or incorrectly encoded payload). The second succeeds — the payload is
correctly URL-encoded and the OR boolean expression evaluates to true. The decoded
URI makes the injected SQL syntax clearly readable.

**urldecode reference**:

```spl
| eval Decoded=urldecode("%27%20OR%20%271%27%3D%271")
-- Results in: ' OR '1'='1
```

---

### 4. SQL Injection Detection — Keyword Method (MadHatter)

MadHatter used a structurally different SQL injection approach — injecting **SQL
keywords and control characters** rather than boolean expressions. Key indicators:

| Indicator | Purpose |
|-----------|---------|
| `SELECT`, `UNION`, `INSERT`, `UPDATE`, `DELETE`, `DROP` | Direct database manipulation commands |
| `;` | Statement separator — allows injecting a second query |
| `--` | SQL comment — causes everything after it to be ignored by the database |

**Example payload**: `'); DROP TABLE Accounts; --`

**Detection query**:

```spl
index=main UserAgent!="HumptyDumpty"
(URI=*SELECT* OR URI=*UNION* OR URI=*DROP* OR URI=*INSERT*
OR URI=*UPDATE* OR URI=*DELETE* OR URI=*;* OR URI=*--*)
```

**Step 2 — Decode to confirm**:

```spl
index=main UserAgent="MadHatter"
| eval Decoded=urldecode(URI)
```

**Expected Outcome**: MadHatter's decoded URI reveals SQL keyword injection with
statement separators and comment strings — a fundamentally different construction
from HumptyDumpty's boolean approach, but targeting the same vulnerability class.

## Common Mistakes

1. **Relying on raw URI values for SQL injection detection**
   - Payloads are typically URL-encoded in transit. `%27` is `'`, `%20` is a space.
   - Always run `urldecode(URI)` before drawing conclusions about payload content.

2. **Using only length thresholds for buffer overflow detection**
   - `length(URI)>100` catches long URIs but includes legitimate long URLs.
   - The parameter-specific regex (`[^&]{100,}`) narrows to input values specifically,
     reducing false positives.

3. **Searching for SQL keywords without URL decoding first**
   - `SELECT` in a raw URI may appear as `%53%45%4C%45%43%54`. Keyword matching
     on the raw field will miss encoded payloads. Decode first, then search.

4. **Treating the two SQL injection methods as the same attack**
   - Boolean injection (`OR '1'='1`) and keyword injection (`DROP TABLE`) have
     different objectives and require different remediation (parameterized queries
     address both, but WAF rules differ by pattern).

5. **Assuming a failed attack attempt is irrelevant**
   - HumptyDumpty's failed first attempt is forensically significant — it reveals
     iterative attacker behavior and confirms intent even if no damage occurred.

## Challenge

1. **Enumerate all attack types in one query**: Build a single search that flags
   events containing null bytes, SQL keywords, or abnormally long parameters,
   and tags each with its attack type using `eval`:
```spl
   index=main
   | eval AttackType=case(
       match(URI,"%00"), "Null Byte Injection",
       match(URI,"(?i)SELECT|UNION|DROP|INSERT|UPDATE|DELETE"), "SQL Injection - Keyword",
       match(URI,"(?i)OR.+\="), "SQL Injection - Boolean",
       len(URI)>100, "Possible Buffer Overflow",
       true(), "Clean"
   )
   | search AttackType!="Clean"
   | stats count by UserAgent, AttackType
```

2. **Timeline the attacks**: Did the attackers operate at the same time or at different
   hours? Use `timechart span=1h count by UserAgent` filtered to the four attacker
   User-Agents to map the attack timeline.

3. **Check for additional encoded payloads**: Some attackers double-encode payloads
   (e.g., `%2500` encodes to `%00` after one decode pass). Apply `urldecode` twice
   and check if new attack patterns surface.
