# Splunk_Brute_Force_Detection_Via_Syslog_Mental_Model

| Concept | Operational Meaning |
|---|---|
| Lower-count sourcetype | The sourcetype with fewer log entries — in this case `syslog`; fewer entries than `pan:traffic` but the entries themselves reveal a distinct attack pattern |
| Syslog as authentication record | Linux/Unix hosts write authentication attempts — both successes and failures — to syslog; this makes it the primary source for detecting brute force against host services |
| Raw field inspection | The `_raw` field contains the full unprocessed log event text — scanning it directly reveals attack patterns not surfaced by field extraction alone |
| Failed login pattern | Repeated authentication failure messages from the same source IP in a compressed timeframe is the defining signature of a brute force attack |
| Brute force definition | Systematic, repeated submission of credential combinations against an authentication service until a valid pair is found |
| High failure count + short timeframe | The combination of volume and velocity distinguishes brute force from legitimate failed logins (which are isolated and infrequent) |
| No valid credentials assumed | Brute force is the method of choice precisely when the attacker does not have valid credentials — distinguishes it from credential stuffing where known pairs are tested |

# Splunk_Brute_Force_Detection_Via_Syslog_Investigation_Checklist

| Step | Action | Notes |
|---|---|---|
| 1 | Confirm time range is set to **All Time** | Consistent with all prior tasks |
| 2 | Filter results to syslog sourcetype only | Either click `extracted_sourcetype` in the field sidebar and select `syslog`, or add `extracted_sourcetype=syslog` directly to the query |
| 3 | Scope to attacker's IP | Retain `src_ip=199.203.100.86` filter from Task 5 |
| 4 | Inspect the `_raw` field on returned events | Raw field contains the unprocessed log text — look for authentication-related keywords |
| 5 | Identify repeated failed login messages | Look for terms such as `failed`, `authentication failure`, `invalid password`, `Failed password` in rapid succession |
| 6 | Assess volume and velocity | High count of failure messages within a short timeframe confirms brute force — not isolated user error |
| 7 | Classify the attack | Repeated failed logins from external IP at high velocity = **Brute Force** |

# Splunk_Brute_Force_Detection_Via_Syslog_Verification_Steps

| Check | Expected Result |
|---|---|
| Syslog filter returns results | Events present for `src_ip=199.203.100.86 extracted_sourcetype=syslog` |
| Raw field contains authentication failure messages | Keywords such as `Failed password`, `authentication failure`, or `invalid user` visible in `_raw` |
| High volume of failure messages | Many repeated failed login entries — not isolated occurrences |
| Short timeframe between attempts | Timestamps show rapid succession — automated tooling signature |
| No successful logins visible at this stage | All syslog events show failure — success confirmation is Task 7 |
| Final classification | **Brute Force** |

# Splunk_Brute_Force_Detection_Via_Syslog_Failure_Checks

| Symptom | Likely Cause | Fix |
|---|---|---|
| No results after adding syslog filter | Sourcetype value case mismatch or typo | Verify exact `extracted_sourcetype` value from Task 5 field sidebar — use exact casing |
| Raw field not visible | Default table view does not include `_raw` | Click individual events to expand full text, or explicitly add `_raw` to a `table` command |
| Cannot identify attack pattern from field extractions alone | Structured fields may not capture all authentication message detail | Always inspect `_raw` directly — it contains the full unprocessed log text |
| Confusing brute force with credential stuffing | Credential stuffing uses known breached pairs — fewer attempts needed | Brute force is characterized by systematic, exhaustive attempts — high volume is the distinguishing factor |
| pan:traffic selected instead of syslog | Higher count sourcetype selected by mistake | Confirm syslog is the **lower**-count sourcetype from Task 5 — that is the one being analyzed here |
