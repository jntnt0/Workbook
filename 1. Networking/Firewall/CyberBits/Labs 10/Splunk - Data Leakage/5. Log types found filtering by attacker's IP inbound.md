# Splunk_Inbound_Attacker_Log_Type_Identification_Mental_Model

| Concept | Operational Meaning |
|---|---|
| Reversing the src/dest filter | Prior tasks filtered on `src_ip=172.16.100.21` (internal host); this task flips to `src_ip=199.203.100.86` (attacker) to view inbound traffic |
| `extracted_sourcetype` field | Identifies the origin log format of each event — reveals which devices observed and recorded the attacker's inbound traffic |
| `pan:traffic` sourcetype | Palo Alto Networks firewall traffic log — generated by the perimeter ASAv/Palo Alto as traffic traverses it |
| Syslog sourcetype | System-level log from a Linux/Unix host — generated when attacker traffic reaches and interacts with a host-level service |
| Windows Event logs absence | Windows Event logs are generated by Windows hosts upon authentication, process execution, or system events — if the attacker has not yet authenticated to a Windows host, no Windows Event logs will appear |
| Log type as attack stage indicator | FW logs confirm the attacker reached the perimeter; Syslog confirms the attacker reached a host — together they indicate active inbound probing beyond the firewall |

# Splunk_Inbound_Attacker_Log_Type_Identification_Investigation_Checklist

| Step | Action | Notes |
|---|---|---|
| 1 | Confirm time range is set to **All Time** | Consistent with all prior tasks |
| 2 | Run base query filtering `src_ip` to attacker's IP | Flips perspective — now viewing traffic initiated by the external attacker inbound |
| 3 | Locate the `extracted_sourcetype` field in results | Found in the field sidebar or within individual event detail panes |
| 4 | Review all distinct `extracted_sourcetype` values | Identify every log source that recorded attacker activity |
| 5 | Classify each sourcetype | `pan:traffic` = Palo Alto FW logs; `syslog` = host-level system logs |
| 6 | Note the absence of Windows Event logs | Confirms attacker has not yet authenticated to or executed on a Windows host at this stage |
| 7 | Record confirmed log types | FW logs (`pan:traffic`) and Syslogs — both required for the answer |

# Splunk_Inbound_Attacker_Log_Type_Identification_Failure_Checks

| Symptom | Likely Cause | Fix |
|---|---|---|
| No results returned | Using `dest_ip` instead of `src_ip` for attacker IP | Inbound attacker traffic has the attacker as `src_ip` — confirm filter direction |
| `extracted_sourcetype` field not visible | Field not auto-extracted in default view | Click on an individual event to expand it and locate the field manually |
| Only one sourcetype visible | Time range too narrow or results not fully loaded | Set to All Time; scroll through full result set |
| Windows Event logs present unexpectedly | Time range includes post-compromise activity from later tasks | Scope time range or cross-reference with Task 6/7 findings to separate attack stages |
