# Splunk_Suspicious_IP_Pair_By_Connection_Count_Mental_Model

| Concept | Operational Meaning |
|---|---|
| `stats count by src_ip dest_ip` | Aggregates all log events into unique source/destination pairs with a total connection count per pair |
| `sort -count` | Sorts results descending — highest connection counts appear first |
| Internal-first filtering | Top results are typically internal-to-internal pairs; these are expected and not inherently suspicious |
| Known infrastructure exclusion | High-count pairs involving known legitimate infrastructure (e.g., DNS servers) must be cross-referenced against the network map before flagging |
| Suspicious external destination | An external IP with an abnormally high connection count that does not appear in the network map is the primary indicator of interest |
| Data exfiltration pattern | A persistent, high-volume connection from an internal host to an unknown external IP is a leading indicator of exfiltration activity |

# Splunk_Suspicious_IP_Pair_By_Connection_Count_Investigation_Checklist

| Step | Action | Notes |
|---|---|---|
| 1 | Set Splunk time range to **All Time** | Results will be empty or misleading if time range is too narrow |
| 2 | Run base query against the data source | Confirm table output populates with `src_ip`, `dest_ip`, `count` columns |
| 3 | Review top results — identify internal pairs | RFC 1918 addresses (`10.x`, `172.16.x`, `192.168.x`) on both sides are expected noise |
| 4 | Identify first external `dest_ip` in the list | Cross-reference against network map |
| 5 | Validate against network map or `dvc_name` field | If the IP belongs to a known device (e.g., DNS server), eliminate it and move to the next |
| 6 | Flag the next external `dest_ip` with no network map presence | This is the suspicious destination |
| 7 | Note the full `src_ip -> dest_ip` pair for downstream analysis | This pair carries forward into byte, port, and log-type queries |

# Splunk_Suspicious_IP_Pair_By_Connection_Count_Exclusion_Logic_Skeleton

Use after identifying a candidate external IP to validate it is not legitimate infrastructure:

```spl
source="<DATA_SOURCE>" src_ip=<CANDIDATE_SRC> dest_ip=<CANDIDATE_DEST> | table dvc_name src_ip dest_ip
```

**Lab-specific instantiation (DNS server validation):**
```spl
source="/home/splunk/data.csv" dest_ip=199.203.100.100 | table dvc_name src_ip dest_ip
```

| Placeholder | Value |
|---|---|
| `<CANDIDATE_DEST>` | IP being evaluated against network map |
| `dvc_name` | Device name field — confirms whether IP belongs to a known network asset |

# Splunk_Suspicious_IP_Pair_By_Connection_Count_Verification_Steps

| Check | Expected Result |
|---|---|
| Query returns a populated table | `src_ip`, `dest_ip`, `count` columns present with multiple rows |
| Top rows are internal pairs | Both src and dest are RFC 1918 addresses |
| First external destination cross-checked | `dvc_name` confirms it is a known asset (e.g., DNS) — not flagged |
| Second external destination has no map presence | No `dvc_name` match, no network map entry — flagged as suspicious |
| Confirmed suspicious pair | `172.16.100.21 -> 199.203.100.86` |

# Splunk_Suspicious_IP_Pair_By_Connection_Count_Failure_Checks

| Symptom | Likely Cause | Fix |
|---|---|---|
| No results returned | Time range not set to All Time | Change time picker to **All Time** and re-run |
| Query syntax error after paste | Encoding issue — quotes pasted as `?` | Manually retype quote characters in Splunk search bar |
| All results are internal pairs | Data set may be filtered or truncated | Scroll further down the results table |
| Cannot find `dvc_name` field | Field not extracted by default | Click on a result row and manually locate the field in the event detail pane |
