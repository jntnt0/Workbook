# Lab 3 — Determining the Entry Point of the Malicious Payload

## Tab 1

Tracing a payload back to its origin requires following the process tree upstream — parent by parent — until you reach the initial user action that started the chain. A malicious binary doesn't appear out of nowhere; something downloaded it, and something triggered that download. The `Image` and `ParentImage` fields in process logs are your roadmap. Walk them backward and the entry point reveals itself.

The goal of this task is to determine how `GoogleUpdate.exe` arrived on the host and what user action was responsible for introducing the payload to the network.

---

### Step-by-Step Thought Process

**Search by Image path, not process name.**  
Searching for `GoogleUpdate.exe` by name alone risks returning results from legitimate Google update processes running from `C:\Program Files\Google\`. You must search by the full `Image` path — `C:\Users\User067\Downloads\GoogleUpdate.exe` - to isolate the malicious instance exclusively.

**Identify the immediate parent.**  
The `ParentImage` field of the malicious GoogleUpdate.exe reveals `Chrome.exe` as the process that spawned it. Chrome doesn't execute binaries on its own ,  it downloads them. This confirms the payload was downloaded via the browser by a user accessing a URL.

**Pivot to Chrome's parent.**  
The investigation doesn't stop at Chrome. Alter the table query to search for the Chrome `Image` and parse using `| table Image ParentImage CommandLine`. Chrome's own `ParentImage` reveals the process that launched it — and that process is the real entry point indicator.

**Read the ParentImage of Chrome.**  
Chrome's parent is `Outlook.exe`. A user received a phishing email, clicked a hyperlink inside it, Outlook launched Chrome to handle the URL, and Chrome downloaded the malicious payload to the Downloads folder. That click is the entry point.

---

### Expected Splunk Queries

**Step 1 — Confirm Chrome as GoogleUpdate's parent:**
```
source=/home/splunk/data.csv Image="C:\\Users\\User067\\Downloads\\GoogleUpdate.exe"
| table Image ParentImage CommandLine
```

**Step 2 — Pivot to Chrome and find its parent:**
```
source=/home/splunk/data.csv Image="C:\\Windows\\System32\\chrome.exe"
| table Image ParentImage CommandLine
```

```
source=/home/splunk/data.csv Image="C:\\Users\\User067\\Downloads\\GoogleUpdate.exe"
```


---

### Findings

**Full Process Chain (Bottom to Top):**

| Process | Role |
|---|---|
| `Outlook.exe` | User clicked a hyperlink in a phishing email |
| `Chrome.exe` | Launched by Outlook to handle the URL — downloaded the payload |
| `GoogleUpdate.exe` | Malicious payload dropped to `C:\Users\User067\Downloads\` |
| `cmd.exe` | Spawned by GoogleUpdate.exe to execute recon commands |

**Answer:** A phishing email containing a hyperlink clicked by the user, causing the payload to be downloaded. `Outlook.exe` as Chrome's parent process is the confirming indicator — the attacker weaponized email to get the user to pull the malware down themselves.
