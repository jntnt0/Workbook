# ASA_CLI_Adding_Rules_To_An_Already_Bound_Access_List_Mental_Model

| Concept | Operational Meaning |
|---|---|
| Already-bound ACL | An ACL that has a live `access-group` binding on an interface; new entries added to it are evaluated immediately against traffic |
| No re-binding required | Running `access-group` again is not needed when adding entries to an ACL that is already bound; the binding persists and picks up new rules automatically |
| Predefined service keywords | The ASA recognizes certain well-known port names directly in access-list syntax; `www` maps to TCP port 80 and can be used in place of `eq 80` |
| `eq www` vs `eq 80` | Both are functionally identical on the ASA; `www` is a predefined keyword and either form is accepted |
| TCP vs ICMP rule behavior | TCP is stateful; the ASA tracks the connection and allows the return traffic automatically; no explicit inbound rule for the return direction is needed |
| Rule ordering | ACL entries are evaluated top to bottom; new entries appended via CLI are added at the bottom of the existing ACL |
| ASDM parity | ASDM would have added the access-group at rule creation time; because it was already done via CLI in the prior task, adding new entries here behaves identically to how ASDM works |

# ASA_CLI_Adding_Rules_To_An_Already_Bound_Access_List_Configuration_Checklist

| Step | Task | Device | Command | Expected Result |
|---:|---|---|---|---|
| 1 | Confirm you are in global config mode | ASAv | Note prompt shows `ciscoasa(config)#` | Ready to add access-list entries |
| 2 | Confirm Workstations ACL is already bound | ASAv | `show running-config access-group` | `access-group Workstations_access_in in interface Workstations` is present |
| 3 | Add HTTP permit rule to Workstations ACL | ASAv | `access-list Workstations_access_in extended permit tcp any any eq www` | Command accepted; rule appended to the existing ACL |
| 4 | Open Chrome on Win_Station | Win_Station | Click Chrome on the desktop | Browser opens |
| 5 | Browse to IIS_Server over HTTP | Win_Station | Navigate to `http://IIS_Server` | IIS default page loads; HTTP traffic is passing through the ASA |
| 6 | Confirm the new rule is in running-config | ASAv | `show running-config access-list` | TCP permit www entry appears under Workstations_access_in |

# ASA_CLI_Adding_Rules_To_An_Already_Bound_Access_List_Skeleton

ciscoasa(config)# access-list Workstations_access_in extended permit tcp any any eq www

! No access-group command needed.
! Workstations_access_in is already bound to the Workstations interface.
! The new rule is active immediately.

# ASA_CLI_Adding_Rules_To_An_Already_Bound_Access_List_Failure_Checks

| Symptom | Likely Cause | Fix |
|---|---|---|
| Page fails to load after adding the rule | ACL is not bound to the interface; access-group was never run | Verify with `show running-config access-group`; run `access-group Workstations_access_in in interface Workstations` if missing |
| Page loads intermittently or times out | DNS resolution for IIS_Server is slow or failing | Try browsing to the IP address directly to isolate DNS from firewall as the cause |
| Rule appears in running-config but hit counter stays zero | Traffic is not reaching the ASA on the Workstations interface | Verify Win_Station is on the correct network segment and the ASA interface is up |
| `www` keyword not accepted | Typo or wrong CLI context | Confirm you are in `ciscoasa(config)#` and that the syntax matches exactly; alternatively use `eq 80` in place of `eq www` |
| Page loads without the rule being added | A prior ASDM-created rule already permits HTTP | Check `show running-config access-list` for existing TCP 80 permit entries |
