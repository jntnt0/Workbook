# ASA_CLI_Access_List_Rules_And_The_Access_Group_Requirement_Mental_Model

| Concept | Operational Meaning |
|---|---|
| access-list entry | Defines a rule (permit or deny) for traffic matching source, destination, and service; exists in config but has no effect on its own |
| access-group | Binds an ACL to a specific interface in a specific direction; this is what makes the rule active |
| Inbound direction | Traffic is inspected as it enters the ASA on that interface; `in interface <name>` is the standard for filtering traffic from a network segment |
| ASDM behavior | ASDM runs both `access-list` and `access-group` automatically when a rule is added via the GUI; the CLI requires both steps manually |
| ACL with no access-group | The rules exist in running-config but the ASA does not evaluate them against any traffic; connectivity behaves as if the rules were never written |
| First rule on a new ACL | When an ACL name has never been bound to an interface, adding entries to it does nothing until `access-group` is run |
| Existing ACL binding | Once an ACL is bound via `access-group`, new entries added to that same ACL name take effect immediately without re-running `access-group` |
| `extended` keyword | Required for ACLs that match on both source and destination; standard ACLs match source only and are not used for interface filtering on the ASA |

# ASA_CLI_Access_List_Rules_And_The_Access_Group_Requirement_Configuration_Checklist

| Step | Task | Device | Command | Expected Result |
|---:|---|---|---|---|
| 1 | SSH into ASAv and escalate to config mode | Win_Station | `ssh admin@mgmt.asav.internal` then `enable` then `configure terminal` | Prompt shows `ciscoasa(config)#` |
| 2 | Add ICMP permit rule for Workstations ACL | ASAv | `access-list Workstations_access_in extended permit icmp any any` | Command accepted; no output indicates success |
| 3 | Add ICMP permit rule for Servers ACL | ASAv | `access-list Servers_access_in extended permit icmp any any` | Command accepted; no output indicates success |
| 4 | Open a second CMD window on Win_Station | Win_Station | Start Menu or Run | New terminal window opens |
| 5 | Attempt to ping IIS_Server | Win_Station | `ping IIS_Server` | Request timed out; ICMP is blocked because no access-group binding exists |
| 6 | Confirm the rules exist in running-config | ASAv | `show running-config access-list` | Both ICMP permit entries are visible but no access-group is shown binding them |

# ASA_CLI_Access_List_Rules_And_The_Access_Group_Requirement_Skeleton

ciscoasa(config)# access-list Workstations_access_in extended permit icmp any any
ciscoasa(config)# access-list Servers_access_in extended permit icmp any any

! Rules are now in running-config but are NOT active.
! Ping to IIS_Server will time out until access-group bindings are added.
! See ASA_CLI_Binding_Access_Lists_To_Interfaces_With_Access_Group for next step.

| Command | Where to Run | What It Confirms |
|---|---|---|
| `show running-config access-list` | `ciscoasa#` | Shows all ACL entries; confirms rules were written but reveals no access-group binding |
| `show access-list` | `ciscoasa#` | Shows ACL entries with hit counters; zero hits confirms traffic is not being evaluated |

# ASA_CLI_Access_List_Rules_And_The_Access_Group_Requirement_Failure_Checks

| Symptom | Likely Cause | Fix |
|---|---|---|
| Ping times out after adding access-list entries | No access-group binding exists for those ACL names | Run `access-group <acl-name> in interface <interface-name>` for each ACL |
| `show access-list` shows zero hit counts | ACL is not bound to any interface; traffic is never evaluated against it | Bind with `access-group` |
| Rule appears in `show running-config` but traffic is unaffected | ACL name exists but has no interface association | Verify with `show running-config access-group`; add binding if missing |
| `show running-config access-list` shows nothing | Command was entered in user EXEC mode or the ACL name was mistyped | Escalate to privileged EXEC and verify ACL name spelling |
| Ping fails with destination unreachable instead of timeout | A downstream device or the ASA is actively rejecting with an ICMP unreachable; different from a silent drop | Check for explicit deny rules or inspect policies that generate unreachables |
