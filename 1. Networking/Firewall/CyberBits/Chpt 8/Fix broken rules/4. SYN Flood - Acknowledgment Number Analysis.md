#### SYN_Flood_Acknowledgment_Number_Analysis_Mental_Model
| Concept | Operational Meaning |
|---|---|
| Acknowledgment number | A TCP header field that tells the sender which byte the receiver expects next |
| ACK number of 0 | No connection has been established. There is nothing to acknowledge, so the field is empty |
| SYN packet ACK behavior | A pure SYN packet always carries an ACK number of 0 because it is the first step of a handshake with no prior data exchange |
| ACK number in established connections | Once a connection is established, the ACK number reflects the next expected sequence byte from the other side |
| Static ACK value | Because SYN flood packets are all pure SYN with no established connection, the ACK number is consistently 0 across every packet |
| Snort ack option | A non-payload TCP option that matches packets based on their acknowledgment number value |
| Detection value of ack:0 | Combined with flags:S, ack:0 tightens the rule to pure SYN packets only, reducing false positives |

# SYN_Flood_Acknowledgment_Number_Analysis_Key_Observations

| Parameter | Value | Static / Dynamic | Snort Rule Field |
|---|---|---|---|
| Acknowledgment number | 0 | Static | ack:0; |
| ACK flag bit | Unset | Static | Consistent with pure SYN packet behavior |
| Sequence number | Varies per packet | Dynamic | Not reliable for Snort detection |
| Flag and ACK relationship | SYN flag set + ACK number 0 = pure SYN | Static combination | flags:S; ack:0; used together |

# SYN_Flood_Acknowledgment_Number_Analysis_Snort_Rule_Skeleton

    alert tcp $EXTERNAL_NET any -> $HOME_NET any
    (msg:"SYN Flood Detected"; flags:S; ack:0;
    sid:1000002; rev:1;)

    Note: Adding ack:0 alongside flags:S tightens detection to pure SYN
    packets only. A legitimate SYN-ACK would have a non-zero ACK number
    and would not trigger this rule. Source IP, destination IP, and source
    port will be added to the rule header once all static parameters are
    confirmed.

# SYN_Flood_Acknowledgment_Number_Analysis_Verification

| Check | What to Look For | Confirms |
|---|---|---|
| Acknowledgment Number field | Value is 0 across all packets #93-#105 | ACK number is a static, reliable detection parameter |
| ACK flag bit | ACK bit is unset in Flags section | Consistent with ACK number of 0, confirms pure SYN behavior |
| Sequence number varies | Different sequence numbers across packets | Sequence number is dynamic and not suitable for Snort detection |
| SYN-ACK packets excluded | SYN-ACK packets outside range carry non-zero ACK numbers | ack:0 correctly excludes legitimate handshake responses |

# SYN_Flood_Acknowledgment_Number_Analysis_Failure_Checks

| Symptom | Likely Cause | Fix |
|---|---|---|
| ACK number shows non-zero value | Packet selected is outside the attack range | Confirm packet number is within #93-#105 |
| Cannot locate Acknowledgment Number field | TCP section not fully expanded | Expand all rows under Transmission Control Protocol |
| ACK flag appears set alongside SYN | Selected packet is a SYN-ACK from normal traffic | SYN-ACK packets are legitimate responses outside the attack range; re-check packet number |
| Acknowledgment number field missing entirely | Viewing a non-TCP packet | Confirm the packet is TCP before expanding the header |
| Confused ACK number with ACK flag | Two separate fields share the ACK concept | ACK flag is a 1-bit field in Flags. ACK number is a 32-bit field in the header. Both are 0 in a pure SYN |
